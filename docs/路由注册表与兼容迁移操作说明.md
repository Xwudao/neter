# 路由注册表与兼容迁移操作说明

## 背景

旧项目的 `HttpEngine` 持有每个具体 Route，并在 `Register()` 中逐一调用 `Reg()`。每新增一个 Route，生成器都必须改动结构体字段、构造函数和注册方法，维护成本高。

新结构将这些具体依赖收敛到 `RouteRegistry`：

```go
type Registrar interface {
	Register(router gin.IRouter)
}

type RouteRegistry []Registrar
```

`HttpEngine` 只依赖 `RouteRegistry`，调用 `routes.RegisterAll(engine)` 完成注册。

## 新项目行为

新模板包含 `internal/routes/registry.go`。执行：

```sh
nr gen route -n example -p v1
```

会创建 Route、更新 `NewRouteRegistry` 的参数和返回列表、更新 `ProviderRouteSet`，并默认执行 `wire ./cmd/app`。批量生成或需要自行控制 Wire 时可加 `--skip-wire`。

## 旧项目迁移

先只预览，不写文件：

```sh
nr migrate routes --dir /path/to/project
```

确认预览的 Route 列表后，再执行：

```sh
nr migrate routes --dir /path/to/project --apply-registry
```

迁移会：

- 创建 `internal/routes/registry.go`；
- 将 Route 的 `Reg()` 改为 `Register()`；
- 将 `HttpEngine` 中的具体 Route 字段和构造参数替换为 `RouteRegistry`；
- 保留 `HttpEngine.Register()` 中不属于 Route 的逻辑；
- 更新 `ProviderRouteSet`；
- 清理迁移后无用的 Route 包导入；
- 默认执行 `wire ./cmd/app`。

命令会先校验项目内 `.Reg()` 调用是否可安全识别为 Route 调用；无法识别时停止，不写入猜测性修改。`--skip-wire` 仅供已明确需要自行处理 Wire 的高级场景使用。

## 与类型化 API 的关系

注册表迁移不改变原有 Handler、`core.WrapData` 或响应结构，因此旧项目迁移后可保持接口行为。

`nr` 分别检测两种能力：

- 存在 `NewRouteRegistry`：新 Route 使用 `Register()` 并自动加入注册表；
- `internal/core/rtn.go` 存在 `NoInput` 泛型包装器：新 Route 才使用 `NoInput/JSON/Request` 类型化 API。

因此仅做注册表迁移的旧项目，后续 `nr gen route` 仍生成兼容的 `Register() + WrapData + WrappedHandlerFunc`，不会要求立即迁移业务 Handler。只有执行过 `--inject-router` 的项目，新增 Route 才会生成 `Register(router gin.IRouter)`，并移除 Route 对 `*gin.Engine` 的构造器依赖。

## Router 注入迁移

普通 API Route 不应把 `*gin.Engine` 保存在结构体或 Wire 构造器中。新模板使用：

```go
func (r *ExampleRoute) Register(router gin.IRouter)
```

`RouteRegistry.RegisterAll(router)` 由 `HttpEngine` 传入自己的 Engine。这样 Route 构造器只保留业务依赖，测试也可以用任意 `gin.IRouter` 完成注册。

已完成注册表迁移的旧项目可先预览，再写入：

```sh
nr migrate routes --dir /path/to/project --inject-router
nr migrate routes --dir /path/to/project --inject-router --apply-registry
```

该迁移只处理普通 Route 的 Engine 字段、构造器参数和 `Register` 调用，不会将全局静态文件、模板、`NoRoute` 等 Engine 级能力搬入 Route。

## 验证

迁移完成后建议依次运行：

```sh
wire ./cmd/app
go build ./cmd/app
go test ./internal/routes/...
```

对 go-hunhepan 的实际迁移已验证：RouteRegistry 已生成、Wire 已重建、`go build ./cmd/app` 通过。

## Biz / Repo 分层约定

HTTP 参数类型继续放在 `internal/domain/params`，保留 `json`、`form`、校验等传输层标签；不要把这些类型直接传到 Biz。Biz 对外接收纯业务 `Command` / `Query`，例如 `CreateUserCommand`、`UserListQuery`，再由 Route 映射参数到业务类型。

分页查询统一使用 `core.Page[T]`：

```go
func (b *UserBiz) List(ctx context.Context, query UserListQuery) (core.Page[*ent.User], error)
```

这样 Repo 仍只负责数据读取与写入，Biz 负责默认值、权限和业务规则，Route 只负责绑定和响应。旧的 `core.NewListRtn` 保留不动，新接口可逐步改为直接返回 `core.Page[T]`。

Biz 遇到预期业务失败时使用 `internal/domain/errs`，不要依赖数据库错误文本：

```go
return nil, errs.NewConflict("名称已存在", err)
```

`core.NewRtnWithErr` 会保留这个稳定的用户提示，并继续兼容原有 Ent 和连接错误的处理。

新建 Biz 时可选择生成基础合约：

```sh
nr gen biz -n user --with-contracts
```

该选项新增 `internal/biz/user_contract.go`，包含 `UserListQuery`、`CreateUserCommand` 与 `UpdateUserCommand`。它不会改写已有 Biz、Repo 或 HTTP 参数文件；`--with-params` 的旧行为保持不变。CRUD 场景可同时使用 `--with-crud --with-iface`，生成的 Biz 接口会包含当前 CRUD 方法，便于 Route 依赖接口并用 mockgen 生成测试替身。

为新 Biz 生成 Repository mock 和测试骨架可执行：

```sh
nr gen mock -n user --with-test
```

该命令不会覆盖已有测试文件；需要先安装 `mockgen`。测试骨架只提供依赖替身和 Arrange / Act / Assert 的起点，具体业务断言仍应由实现者补上。
